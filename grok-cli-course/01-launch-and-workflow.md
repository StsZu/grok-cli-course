# Урок 1 — Запуск і workflow

## Про що цей урок

Перший контакт з Grok CLI: перевірка встановлення, запуск TUI, довідка по прапорцях, безпечний старт у git-репозиторії та command palette (`/`).

**Головна мета — не вивчити всі команди Grok CLI, а навчитися швидко знаходити потрібну команду, розуміти її ризик і застосовувати її в реальному сценарії.**

---

## Перед стартом

Grok CLI — це TUI (термінальний інтерфейс) від xAI. Агент читає файли проєкту, викликає tools і може змінювати код. Тому **завжди запускай його в папці проєкту**, а не в `~` або `/`.

Корисна звичка: `git status` перед `grok` — бачиш, що вже змінено, і не змішаєш свої правки з правками агента.

---

## Команди

### `which grok`

| | |
|---|---|
| **Що робить** | Показує шлях до бінарника Grok CLI |
| **Коли** | Після встановлення; якщо `grok` не знаходиться в PATH |
| **Приклад** | `which grok` |
| **Результат** | `/opt/homebrew/bin/grok` або `/usr/local/bin/grok` |
| **Помилки** | Порожній вивід — Grok CLI не встановлено або не в PATH |
| **Ризик** | **Низький** |

---

### `grok`

| | |
|---|---|
| **Що робить** | Запускає Grok CLI TUI в поточній папці |
| **Коли** | Початок роботи з агентом над проєктом |
| **Приклад** | `cd ~/Projects/demo && grok` |
| **Результат** | Відкривається TUI; prompt `grok>`; `Shift+Tab` — перемикання session modes |
| **Помилки** | `command not found` — встанови CLI з [x.ai/cli](https://x.ai/cli); помилка логіну — потрібен xAI акаунт |
| **Ризик** | **Середній** — агент може змінювати файли після підтвердження tool calls |

---

### `grok --help`

| | |
|---|---|
| **Що робить** | Довідка по прапорцях запуску |
| **Коли** | Перед незвичним запуском (наприклад `--always-approve`) |
| **Приклад** | `grok --help` |
| **Результат** | Список опцій: `--always-approve`, `--help` тощо |
| **Помилки** | — |
| **Ризик** | **Низький** |

---

### `git status`

| | |
|---|---|
| **Що робить** | Показує змінені, staged і untracked файли в репозиторії |
| **Коли** | **Перед** `grok` і **після** сесії з агентом |
| **Приклад** | `git status` |
| **Результат** | `modified: src/app.ts`, `nothing to commit, working tree clean` |
| **Помилки** | `not a git repository` — не в git-проєкті; для агента це ОК, але немає diff для відкату |
| **Ризик** | **Низький** |

---

### `/`

| | |
|---|---|
| **Що робить** | Відкриває command palette — список slash-команд TUI |
| **Коли** | Забув назву команди; хочеш побачити доступні дії в сесії |
| **Приклад** | У TUI введи `/` і Enter |
| **Результат** | Список: `/new`, `/context`, `/plan`, `/quit`… |
| **Помилки** | Працює лише **всередині TUI**, не в звичайному shell |
| **Ризик** | **Низький** |

---

## Сценарій: безпечний перший запуск

```bash
cd ~/Projects/demo
git status
which grok
grok
```

У TUI:

```
/
```

Обери з palette `/context` або `/session-info` — переконайся, що сесія стартувала в правильній папці.

Постав задачу текстом (звичайний промпт), наприклад: «Поясни структуру проєкту, не змінюй файли».

---

## Command palette vs промпт

| Тип вводу | Де | Приклад |
|-----------|-----|---------|
| Slash-команда | TUI | `/context`, `/plan` |
| Промпт | TUI | «Додай тест для функції X» |
| Shell-команда | Термінал (поза TUI) | `git status`, `grok` |

Slash-команди — швидке керування сесією. Промпти — робота з кодом і логікою.

---

## Практичне завдання

1. Перейди в git-репозиторій. Виконай `git status` — зафіксуй, чи є незакомічені зміни.
2. `which grok` — запиши шлях до бінарника.
3. `grok --help` — знайди прапорець `--always-approve` (поки **не** використовуй).
4. Запусти `grok`. Введи `/` — перелічи 5 slash-команд, які бачиш.
5. Вийди через `/quit`. Знову `git status` — порівняй з кроком 1.
6. **Тренажер:** відкрий [тренажер](../index.html), пройди модуль «1. Запуск і workflow».
